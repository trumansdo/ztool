# iced的3d程序概要

有以下实质接口：
- scene 3d程序的唯一对外的接口，名称就是场景，放置视图元素，但数据是cpu端的数据结构，以面向对象的逻辑设计
- primitive 原始数据接口，数据是面向gpu的数据结构，以gpu并行逻辑设计
- pipeline 连接scene与primitive的管道，主要功能是上传primitive原始数据，与渲染逻辑

对于3d程序又有以下概念：
- 相机数据结构：用于处理MVP矩阵中的vp计算；m是模型自身的线性变换，依赖线性代数，参考3blue1brown的《线性代数的本质》。相机只管vp，不管m，m由每个模型自己决定
- 顶点数据结构：顶点存的是局部空间的原始位置，不是世界坐标，所有东西都围着一个原点建模。静态vbo里可以放这些数据：顶点坐标（position）、法线方向（normal）、纹理坐标（uv），有时候还有颜色（color）、切线（tangent）之类的，具体放什么看shader需要什么，不是固定的。这些是所有实例共用的，gpu拿到后，用实例变换矩阵一乘，才变成实际世界位置。所以实际位置不存在缓存里，是gpu实时算出来的
- 统一变量数据结构：共享变量，比如光源位置、颜色、时间之类的，所有顶点和片元都能用，不用每个模型各自传一份。相机也是统一变量的一种，传的是vp矩阵

实现步骤：

1. iced::widget::shader::Program
3d程序总入口，主要实现draw方法完成渲染

2. iced::widget::shader::Primitive
上传GPU端的原始数据：顶点缓冲（vbo）、索引缓冲（ibo）

3. iced::widget::shader::Pipeline
联通CPU端数据结构与GPU端byte数据的管道。绑定shader、顶点布局、混合模式这些渲染状态。需要自己实现trait，不是自动生成的。
我的理解是等价于Java里面的一些全局数据/变量，如顶点缓冲/矩阵缓冲/纹理/实际渲染管线

4. 基于以上三个接口需要实现CPU端的以下四样：
- shape：场景里的实体，对应scene，管模型的线性变换（m矩阵）
- 顶点数据：对应primitive里的vbo/ibo
- 相机：对应统一变量，管vp矩阵
- 统一变量：管光源、颜色之类的共享数据

外加做一堆转换工作：cpu结构体转byte、算mvp矩阵、构造实例变换矩阵这些线性代数的活。

5. iced大致调用流程

首帧：
Program::draw          → （可能用于测量尺寸）
       ↓
Pipeline::new          → 一次（创建 GPU 资源）
       ↓
Primitive::prepare     → 每帧（上传数据到 GPU）
       ↓
Primitive::draw        → 每帧（返回 false）
       ↓
Primitive::render      → 每帧（自建离屏 RenderPass 绘制）

后续每帧：
Program::draw          → 每帧（返回 Primitive）
       ↓
Primitive::prepare     → 每帧（上传数据到 GPU）
       ↓
Primitive::draw        → 每帧（返回 false）
       ↓
Primitive::render      → 每帧（自建离屏 RenderPass 绘制）


开发iced的shader流程
- 实现shader::Program trait
  - 需要实现Primitive
- 实现shader::Primitive trait
  - 需要实现Pipeline
- 实现shader::Pipeline trait

三者是联动的，最终目标是实现shader::Primitive中的prepare和render方法。
- prepare方法是每次都要上传本次调用的管线GPU数据，如Uniform全局向量/投影矩阵/旋转等线性变换矩阵
- render方法是执行GPU绘制配置，目的是实现自定义RenderPass，因为绝大部分需要深度缓冲区
- draw方法也是执行GPU绘制配置，也既是默认的RenderPass，只是没有深度缓冲


实际上只需要实现上面的内容，但为了更好的工程化代码，
GPU数据的准备和处理需要用更优雅的面向对象设计
